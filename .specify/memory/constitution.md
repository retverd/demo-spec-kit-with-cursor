# Конституция проекта: demo-spec-kit-with-cursor

**Версия:** 1.1.0  
**Дата ратификации:** 2025-12-02  
**Последнее изменение:** 2025-12-04

## Обзор

Данная конституция определяет принципы и стандарты разработки для консольного проекта, обеспечивающие высокое качество кода, повторяемость генерации и надежное тестирование.

## Принципы

### Принцип 1: Качество кода

**Описание:**  
Весь генерируемый код должен соответствовать высоким стандартам качества. Код должен быть читаемым, поддерживаемым и следовать установленным соглашениям о стиле. Используются современные практики разработки, включая принципы SOLID, DRY (Don't Repeat Yourself) и KISS (Keep It Simple, Stupid).

**Требования:**

- Код должен быть структурирован логически и модульно
- Имена переменных, функций и классов должны быть понятными и отражать их назначение
- Сложные функции должны быть разбиты на более мелкие, понятные части
- Комментарии должны объяснять "почему", а не "что" (код сам объясняет "что")
- Избегать дублирования кода через переиспользование и абстракции

**Обоснование:**  
Высокое качество кода снижает технический долг, упрощает поддержку и расширение функциональности, а также уменьшает вероятность появления ошибок.

---

### Принцип 2: Повторяемость генерации

**Описание:**  
Процесс генерации кода должен быть полностью воспроизводимым и детерминированным. При одинаковых входных данных и условиях должны генерироваться идентичные результаты.

**Требования:**

- Все шаги генерации должны быть документированы и автоматизированы
- Использовать версионирование для всех зависимостей и инструментов
- Генерация должна быть идемпотентной (повторный запуск не должен изменять корректно сгенерированный код)
- Логирование всех этапов генерации для возможности аудита
- Использовать seed-значения для случайных генераторов, если они применяются

**Обоснование:**  
Повторяемость обеспечивает предсказуемость результатов, упрощает отладку и позволяет уверенно воспроизводить процесс генерации в различных окружениях.

---

### Принцип 3: Стандарты тестирования

**Описание:**  
Тестирование является обязательной частью процесса разработки. Все тесты должны быть написаны в соответствии с установленными стандартами и практиками.

**Требования:**

- Использовать единый фреймворк для тестирования, выбранный для проекта
- Тесты должны быть изолированными и независимыми друг от друга
- Тесты должны выполняться быстро и не требовать сложной настройки окружения
- Тестовые данные должны быть явными и понятными
- Тесты должны проверять как успешные сценарии, так и граничные случаи и ошибки
- Использовать понятные имена тестов, отражающие проверяемый сценарий

**Обоснование:**  
Стандартизированное тестирование обеспечивает единообразие подходов, упрощает поддержку тестов и повышает уверенность в корректности реализации.

---

### Принцип 4: Покрытие тестами бизнес-функций

**Описание:**  
Все важные бизнес-функции проекта должны быть покрыты тестами. Это гарантирует, что ключевая функциональность работает корректно и регрессии будут обнаружены на раннем этапе.

**Требования:**

- Каждая важная бизнес-функция должна иметь как минимум один тест
- Тесты должны проверять корректность работы функции в различных сценариях
- Критичные бизнес-процессы должны иметь интеграционные тесты
- Покрытие тестами должно отслеживаться и поддерживаться на приемлемом уровне
- При добавлении новой бизнес-функции тесты должны создаваться одновременно с реализацией

**Обоснование:**  
Покрытие тестами важных бизнес-функций защищает от регрессий, обеспечивает уверенность при рефакторинге и служит живой документацией ожидаемого поведения системы.

---

### Принцип 5: Документирование и прозрачность

**Описание:**  
Проект должен иметь актуальную документацию, объясняющую архитектуру, процессы и ключевые решения.

**Требования:**

- README должен содержать инструкции по установке, настройке и запуску проекта
- Важные архитектурные решения должны быть задокументированы
- API и публичные интерфейсы должны иметь документацию
- Изменения в конституции и принципах должны отражаться в истории проекта

**Обоснование:**  
Хорошая документация упрощает онбординг новых разработчиков, снижает время на понимание кодовой базы и обеспечивает преемственность знаний.

---

### Принцип 6: Русский язык для спецификаций и комментариев

**Описание:**  
Для удобства работы команды все спецификации проекта и все комментарии в коде должны быть на русском языке.

**Требования:**

- Все спецификации в директории `specs/` должны быть написаны на русском языке
- Все комментарии в исходном коде должны быть на русском языке
- Документация проекта (README, планы, исследования) должна быть на русском языке
- Имена переменных, функций и классов должны быть на английском языке (следуя общепринятым практикам программирования)
- Сообщения об ошибках и логи для пользователей должны быть на русском языке
- Минимизируй использование англицизмов общего назначения, допускается использовать только для специализированных терминов

**Обоснование:**  
Использование русского языка для спецификаций и комментариев обеспечивает единообразие коммуникации в команде, упрощает понимание требований и кода для всех участников проекта, снижает языковые барьеры и ускоряет адаптация новых разработчиков.

## Управление и соблюдение

### Процедура внесения изменений

1. **Предложение изменений:** Изменения в конституцию предлагаются через обсуждение или issue
2. **Обзор:** Предложенные изменения должны быть рассмотрены и одобрены
3. **Версионирование:** Версия конституции обновляется согласно правилам семантического версионирования:
   - **MAJOR** (X.0.0): Несовместимые изменения, удаление или переопределение принципов
   - **MINOR** (0.X.0): Добавление новых принципов или существенное расширение существующих
   - **PATCH** (0.0.X): Уточнения формулировок, исправления опечаток, незначительные улучшения
4. **Обновление зависимых артефактов:** После изменения конституции должны быть обновлены связанные шаблоны и документация
5. **Фиксация:** Изменения фиксируются в системе контроля версий с описательным сообщением коммита

### Политика версионирования

- Версия конституции указывается в формате `MAJOR.MINOR.PATCH`
- При каждом изменении обновляется поле "Последнее изменение"
- История изменений должна отражать причины и обоснование модификаций

### Ожидания по соблюдению

- Все участники проекта должны быть ознакомлены с принципами конституции
- Код-ревью должно проверять соответствие принципам
- Нарушения принципов должны обсуждаться и исправляться
- Регулярные обзоры соблюдения принципов (рекомендуется при каждом значительном изменении)
